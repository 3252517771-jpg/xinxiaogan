## 一、技术框架推荐

### 1.1 核心选型

| 层级 | 推荐 | 版本 | 理由 |
|------|------|------|------|
| Web框架 | **FastAPI** | 0.115+ | 异步原生，自动生成 OpenAPI 文档，Pydantic 集成 |
| ASGI服务器 | **Uvicorn** | 0.30+ | FastAPI 官方推荐的 ASGI 服务器 |
| 生产部署 | **Uvicorn + Supervisor** | — | 单进程 + 守护进程，轻量够用 |
| ORM | **SQLAlchemy 2.0** | 2.0+ | 异步支持成熟，PostgreSQL/SQLite 切换无痛 |
| 数据库驱动 | **asyncpg**（生产）+ **aiosqlite**（开发） | — | SQLAlchemy async engine |
| 迁移工具 | **Alembic** | 1.13+ | SQLAlchemy 官方迁移工具 |
| 认证 | **python-jose + passlib[bcrypt]** | — | JWT 签发验证 + bcrypt 密码哈希 |
| 数据验证 | **Pydantic v2** | 2.x | FastAPI 内置，请求/响应校验 |
| 测试 | **pytest + httpx** | — | FastAPI TestClient 用 httpx |
| 环境管理 | **python-dotenv** | — | `.env` 配置分离 |
| 🆕 机器学习 | **scikit-learn + joblib** | 1.6+ | 风险预测模型训练与本地推理 |
| 🆕 大模型调用 | **openai** (DeepSeek兼容) | latest | 个性化建议 API 调用 |
| 🆕 定时任务 | **APScheduler** | 3.10+ | 推送定时触发（APScheduler替代Celery） |
| 🆕 HTTP 客户端 | **httpx** | 0.27+ | Server酱推送 / 外部 API 异步请求 |

### 1.2 FastAPI 框架自带能力 vs. 需要额外接入

| 能力 | FastAPI 自带？ | 说明 |
|------|:------------:|------|
| HTTP路由 + 方法注解 | ✅ 原生 | `@app.get()`, `@app.post()` |
| 请求参数校验 | ✅ 原生 | Pydantic model 自动校验 |
| 响应模型序列化 | ✅ 原生 | `response_model=` 自动转换 |
| OpenAPI/Swagger文档 | ✅ 原生 | 自动生成 `/docs` 和 `/redoc` |
| CORS中间件 | ✅ 原生 | `CORSMiddleware` |
| 异步支持 | ✅ 原生 | `async def` 原生 async/await |
| 依赖注入 | ✅ 原生 | `Depends()` 可注入 DB session / 认证 |
| 异常处理 | ✅ 原生 + 需自定义 | HTTPException + 自定义 Exception Handler |
| 数据库 ORM | ❌ 需 SQLAlchemy | FastAPI 不内置 ORM |
| 数据库迁移 | ❌ 需 Alembic | 必须额外配置 |
| JWT认证 | ❌ 需 python-jose | FastAPI 无内置 JWT |
| 密码哈希 | ❌ 需 passlib + bcrypt | FastAPI 无内置 |
| 日志 | ❌ 需 logging 或 loguru | FastAPI 无内置日志框架 |
| 限流/速率控制 | ❌ 需 slowapi 或 nginx | 生产环境建议 Nginx 层做 |
| 任务调度/定时器 | ❌ 需 APScheduler/Celery | 🆕 v2.0 引入 APScheduler（推送定时任务） |
| 缓存 | ❌ 需 redis-py / aioredis | 后续版本引入 |

## 三、后端子模块职责边界

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   routers/    │────▶│  services/   │────▶│   models/    │
│  (路由分发)    │     │  (业务逻辑)   │     │  (ORM模型)    │
│  薄层，只做:   │     │  核心逻辑:    │     │  数据持久化   │
│  ✅ 参数提取   │     │  ✅ 评分计算   │     │              │
│  ✅ 调用service│     │  ✅ 趋势聚合   │     │              │
│  ✅ 返回响应   │     │  ✅ 认证逻辑   │     │              │
│  ❌ 不写业务   │     │  ✅ 布局处理   │     │              │
└──────┬───────┘     └──────┬───────┘     └──────────────┘
       │                    │
       ▼                    ▼
┌──────────────┐     ┌──────────────┐
│  schemas/    │     │ middlewares/ │
│  (数据校验)   │     │  (横切关注点) │
│  Pydantic    │     │  JWT验证     │
│  请求/响应    │     │  异常处理    │
└──────────────┘     └──────────────┘
```

**禁止：**
- routers 直接调用 models — 必须通过 services
- services 直接返回 Response 对象 — 只返回数据，由 routers 包装
- schemas 出现业务逻辑 — 只做类型校验和转换

## 五、认证与权限体系

### 5.1 JWT 流程

```
注册 → 密码 bcrypt hash → 存入 users.password_hash
登录 → 查 users → bcrypt verify → 签发 JWT（payload: {sub: user_id, exp}）
后续请求 → Authorization: Bearer <token> → 中间件验证 → 注入当前用户

JWT 配置（.env）：
  SECRET_KEY         = <64位随机字符串>
  ALGORITHM          = HS256
  ACCESS_TOKEN_EXPIRE_MINUTES = 1440（24小时）
```

### 5.2 权限分级

| 级别 | 标签 | 可访问 | 说明 |
|------|------|--------|------|
| 0-公开 | `Public` | /auth/register, /auth/login | 无需token |
| 1-普通用户 | `User` | 所有健康数据API, /user/profile | 只能操作自己的数据 |
| 2-管理员 | `Admin` | /admin/*, /layout/* | 验证 admin_password |

**数据隔离规则：**
```
# auth_middleware.py（伪代码）
user_id = decode_token(token).sub
# 所有健康数据API自动注入 user_id
# 查询SQL都带 WHERE user_id = current_user.id
# 禁止跨用户访问
```

### 5.3 依赖注入实现

```python
# app/utils/dependencies.py
async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db)
) -> User:
    """验证JWT，返回当前用户"""
    payload = decode_jwt(token)
    user = await db.get(User, payload["sub"])
    if not user or user.deleted_at:
        raise HTTPException(401, "无效token")
    return user

async def require_admin(
    current_user: User = Depends(get_current_user)
) -> User:
    """管理员权限验证"""
    if not current_user.is_admin:
        raise HTTPException(403, "需要管理员权限")
    return current_user
```

## 七、异常处理统一规范

```json
// 成功

  "code": 200,
  "message": "ok",
```

### 7.2 HTTP 状态码使用规范

| 状态码 | 触发条件 | 消息模板 |
|--------|---------|---------|
| 200 | 成功 | "ok" |
| 400 | 参数校验失败 | "字段xxx: 具体原因" |
| 401 | 未登录/token过期/无效token | "未授权，请重新登录" |
| 403 | 已登录但无权限 | "需要管理员权限" |
| 404 | 资源不存在 | "资源不存在" 或 "当天无记录" |
| 409 | 资源冲突（如用户名重复） | "用户名已存在" |
| 422 | 请求体格式错误 | "请求格式错误: 具体原因" |
| 429 | 请求频率过高 | "请求过于频繁，请稍后再试" |
| 500 | 服务器内部错误 | "服务器内部错误"（不暴露细节） |

## 八、关键业务逻辑详解

### 8.1 评分计算时序

```
POST /api/health/sleep
  → router 接收 + Pydantic 校验
  → service.calc_sleep(record):
      1. 根据入睡时间、时长、质量、中断数逐项打分
      2. 返回 raw_score (0-100)
  → repo.upsert_sleep_record(user_id, record_date, data, score)
  → repo.upsert_daily_score(user_id, record_date, {sleep_score: score})
  → service.recalc_overall(user_id, record_date):
      1. 读取当前 daily_score 全部5维分
      2. 取有值的维度 AVG → overall_score
      3. 写回 daily_scores.overall_score
  → 返回 {score, overall_score}
```

### 8.2 评分公式（与数据库设计.md一致）

| 维度 | 计算逻辑 |
|------|---------|
| 作息 | 入睡时间(10) + 时长(40) + 质量(40) + 中断(20)，封顶100 |
| 饮食 | 三餐完整度(40) + 热量合理(40) + 多样性(20) |
| 运动 | 达标时长(40) + 步数(30) + 心率(30)，强度加权 |
| 压力 | 压力(40,反向) + 焦虑(30,反向) + 情绪(30) |
| 风险 | 6项体征各15-20分，医学范围判定 |
| 综合 | AVG(有数据的维度) |

### 8.3 趋势数据聚合

```sql
-- GET /api/health/trend 对应的 SQL 伪代码
SELECT
    score_date,
    sleep_score,
    diet_score,
    exercise_score,
    stress_score,
    risk_score,
    overall_score
FROM daily_scores
WHERE user_id = :user_id
  AND score_date BETWEEN :start_date AND :end_date
ORDER BY score_date ASC
```

后端不做复杂聚合，直接返明细，交给前端 Recharts 渲染。
